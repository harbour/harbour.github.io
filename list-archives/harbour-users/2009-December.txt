English readers: Excuse me that I will write in spanish the message, but 
it may help another spanish speakers


Mi experiencia es bastante nueva en harbour, no he experimentado la 
parte gr?fica, lo que si he probado es la compilada nativa de codigo 
clipper, obteniendo aplicaciones consola pero son nativas para el 
sistema operativo que estes usando.
Hay ejemplos en los contribs para cosas graficas.

Lo que tenes que hacer para compilar el codigo clipper es agregarle al 
prg principal
#Esto primero no es estrictamente necesario pero si como yo  tenes el 
clipper en version espa?ol, lo vas a precisar porque el harbour por 
defecto esta en ingles, y no son compatibles las paginas de codigo
# lo cual te va a dar problemas de indices.

REQUEST HB_LANG_ES
REQUEST HB_CODEPAGE_ES850C

PROCEDURE MAIN ()

#Esto por lo mismo
HB_SETCODEPAGE('ES850C')
HB_LANGSELECT('ES')


harbour compila el codigo a C, y luego precisas un compilador de C para 
que te genere el exe.
Te podes instalar el mingw (C gnu) o BCC, ambos gratuitos, el mingw 
aparte es software libre.
Tambien instalado el harbour.
En los paths tienen que estar el path al bin del C instalado , asi como 
el path al bin del harbour.
Creo que la version estable que te podes bajar tiene un dramita con la 
pagina de codigo 850, tendrias que usar una version beta.


Con hbmk2 creas el ejecutable.- Es una utilidad de make...
Tienes que armarte un archivo .hbp en el cual pones los prgs que vas a 
compilar y con el primer prg obtiene el nombre del primer prg, sino 
ejecutando hbmk2 sin parametros podes ver como usarlo mejor con 
parametros para cambiar el nombre del ejecutable, usar librerias, etc.

Guillermo adicionalmente a lo comentado por Maurizio te puedo decir que la
sintaxis del comando

hbmk2 prg1 prg2 prgx  -llibreria -lotralibreria

es una idea  de como funciona , podes ver un monton de ejemplos de como
usarla en los fuentes de harbour

Tambien podes crear una lista    llamada por ejemplo  tuprg.hbp  y llamarla
hbmk2 tuprg.hbp

donde pones los nombres de los prg a incluir como asi tambien

las librerias a incluir.

ejecutando hbmk2 sin modificadores te dara una lista de opciones del mismo

Deberas buscar cuales de las librerias de harbour son las que cubren tus
necesidades

la mayoria de lo habitual de clipper esta cubierto.


Algo que te quiero aclarar es que , harbour no convierte tu programa en modo
grafico sino en 32 bits
si necesitas programar en modo grafico deberas elegir alguna de las
librerias adicionales y ver
como se usan los comandos para las ventanas y demas controles.

Por ejemplo  OOHG , MINIGUI , HMG  , HBQT , HBXBP y hay mas , las primeras
son a mi criterio

las mas faciles de usar aunque son solo para windows y las dos ultimas son
multiplataforma.

*English*

Guillermo in addition to the comments of Maurizio I can tell you that the
syntax is

hbmk2 prg1 prg2 prgx -llibreria -lotralibreria

is an idea of how it works , you a can see a lot of samples in Harbour
sources

You can also create a list and call it , tuprg.hbp eg: hbmk2 tuprg.hbp

where you put the names of the prg to include as well also

the libraries to include.

Running hbmk2 without modifiers will give you a list of options

Need to find which of the harbour are the libraries that cover your needs

most usual of clipper is covered.


Something that I want to clarify is that, harbour does not make your program
in graphical mode but in 32-bit
if you need to program the GUI need to pick any of the additional libraries
and see
commands as used for windows and other controls.

For example ooHG, MiniGUI, HMG, HBQT, HBXBP and there are more, the former
are in my opinion

the easiest to use but are only for windows and the last two are cross
platform.

I hope this information is useful.

Yes it is. In terminal GTs like GTTRM, GTCRS or GTSLN
   SetBlink(.f.)
disable using BLINK attribute in output so '*' in color strings
has no effect.

> I read on xHarbour guide that when we SetBlink() to .f. the asterisk
> effect is bright background.

Some descriptions in xHarbour guide are not precise enough.
Above it's standard Clipper behavior which works in programs
having direct access to video adapter registers. In [x]Harbour
only GTDOS has direct access to hardware and works exactly in
such way. In graphics GTs like GTXWC or GTWVT SetBlink() settings
are ignored and these GTs always highlight background color when
'*' is used in color strings and as I said in remote terminal GTs
(GTTRM, GTCRS or GTSLN) SetBlink(.f.) only disable using blink
attribute. GTWIN ignores this setting and the final behavior
depends on MS-Windows (in window mode it highlights background
and in full screen mode it depends on the video adapter registers
initialized by Windows for full screen text output. In OS2 there
is special system API call which allow to control it in console
window and GTOS2 uses it so it works like GTDOS (and Clipper)
just like described in xHarbour guide.

> I tried the code on several terminal
> emulators, and the result is:
> Terminal Emulator       SetBlink(.t.)        SetBlink(.f.)
> ----------------------------------------------------------
> gnome-terminal          no-effect            no-effect
> xterm                   blinking             no-effect
> rxvt                    bright-background &  no-effect
>                         bright-text
> linux-console           blinking             no-effect
>
> I'm using Harbour 1.1.0-dev-nightlybuild. Is there any workaround on the
> code or settings on the terminal to have a bright background?

There is no standard way to control remotely the behavior of BLINK
attribute in different terminals. From popular terminals only
PuTTY/PTerm (PTerm is stand alone terminal in *nix PuTTY ports)
has escape sequence to control its behavior remotely. Some very
old hardware terminals dedicated for SCO systems also had some
escape sequences which allows to switch the blink bit in used
hardware but it's not portable solution and termcap or terminfo
do not have any entries for it so it can be only hardcoded inside
the code and activated when type of remote terminal is well known
(TERM envvar is not enough here). If you are using PuTTY (or PTerm)
then you can use:
   fwrite( 1, chr( 27 ) + "[=0E" )  /* disable blinking, highlight bkg */
   fwrite( 1, chr( 27 ) + "[=1E" )  /* enable blinking */
in your code.
Anyhow in many terminals the behavior of BLINK attribute can be
changed on user side. Just look at configuration switches.
In Linux console if you are using frame buffer then BLINK attr
always highlights bkg color. If you are using pure VGA text mode
then it can be changed by some programs with root privileges which
can switch blink bit in hardware video adapter registers, i.e.
SVGATextMode has an option to control it (look for 'option "16color"'
in configuration file).
